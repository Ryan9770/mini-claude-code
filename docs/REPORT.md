## 요약 — 세 줄 결론

- **[진단]** 최상위 병목은 추론이 아니라 **“재현 충실도”**였다. 약한 모델은 생각을 못 한 게 아니라, 기존 코드를 정확히 옮겨 적지 못해(토큰 오염) 무너졌다.
- **[모델]** 양자화를 **Q4 → Q8**로 올리자 그 병목이 실질적으로 사라졌다(측정 확인). 재현-오염이 촉발하던 스파이럴이 멈췄다.
- **[하네스]** 재현을 **도구가 다루는 축(①)**은 하네스로 우회·복구했고(AST 편집·경로 수리), **모델 머릿속 축(②)**은 모델만 고친다는 걸 측정으로 분리했다.
- **[방법]** 여러 “그럴듯한 개입”을 **측정으로 폐기**했다(repfix 등). 이 프로젝트의 자산은 기능 개수가 아니라 **증거**다.

## 핵심 진단 — 재현 충실도라는 병목 (①/② 프레임)

직접 측정으로, 실패는 두 곳의 “재현(기존 내용을 그대로 되뱉기)”에서 일어남을 확인했다. 둘은 원인도 해법도 다르다. 새 내용을 *생성*하는 건 약한 모델도 잘한다 — 문제는 이미 있는 걸 *재현*할 때다.

| | ① 도구 호출 안의 재현 | ② 사고 안의 재현 |
|---|---|---|
| **무엇** | edit_file의 old_string, 파일 경로 | 추론 중 소스 재인용 → 스파이럴 |
| **증상** | `total`→`총`, `.md`→`.py` 토큰 오염 | 변수명 환각(`nums`→`params`→…) 후 자기 불신·반복 |
| **해법** | **하네스로 우회 가능** — AST 편집(이름만 지정), 경로 퍼지 수리 **(입증됨)** | **하네스가 못 건드림** — 모델(Q8/QAT)만 해결 |

## 측정 결과 (1) — Q4 vs Q8

동일 프로브를 서버 모델만 바꿔 재측정. Q4가 재현-오염으로 죽던 자리를 Q8은 깨끗이 통과했다.

| 프로브 | Q4 (UD-Q4_K_M) | Q8 (UD-Q8_K_XL) |
|---|---|---|
| 기존 파일 편집 (plain) | ❌ 실패 · 오염→스파이럴 | ✅ 성공 · 오염 0 |
| /language-tutor 입력파일 | ❌ 깨짐 (언구 · レベル · .py) | ✅ 완벽 · 오염 0 |
| /language-tutor 파이프라인 | ❌ 파라리시스로 중단 | ✅ 00–07 완주 · 게이트 준수 |

## 측정 결과 (2) — AST 편집 프로브 (①/② 프레임의 직접 확증)

메서드 편집 과제로 **AST 사용** vs **edit_file 강제**를 대조. pass rate는 고분산이라 무승부로 보여도, 저분산 *메커니즘 지표*가 AST의 기여를 선명히 드러낸다.

| 프로브 · 모델 | AST pass | edit_file pass | ① 편집매칭 실패 (AST vs edit_file) | 병목 |
|---|---:|---:|---:|---|
| 단순 메서드 · Q8 | 3/3 | 3/3 | 0 vs 0 | 없음 |
| 단순 메서드 · Q4 | 3/3 | 3/3 | 0 vs 0 | 없음 |
| 큰 메서드 재작성 · Q4 | 0/3 | 0/3 | **0 vs 2–3** | ② 스파이럴 |

**읽기:** 큰 편집에서 AST는 ①(편집매칭 실패)을 **0으로 제거**했지만(edit_file은 2–3회), pass는 0:0 — **②가 둘 다 죽였다**. 즉 Q8 미만·복잡 편집에서 **AST는 필요조건이지 충분조건이 아니다**. ②는 모델 몫.

## 구축한 것 — 하네스 구성요소

판정: ✅ 검증됨(측정/테스트로 입증) · 🟡 보험(정확성 입증, 발생빈도 미측정) · ❌ 폐기(측정 후 제거)

| 구성요소 | 하는 일 | 판정 | 커밋 |
|---|---|---|---|
| **critic 루프** (컨텍스트 격리) | code→검증게이트→review→수정을 격리 컨텍스트로 반복. 재현 스파이럴을 짧은 컨텍스트로 *가려* 성공률을 올림. | ✅ | — |
| **AST 편집** (`patch_ast_node`) | 심볼 이름으로 함수/클래스/**메서드**를 통째 교체 — old_string 재현 불필요. 이번에 `Class.method`까지 확장. | ✅ | `bcd62ea` |
| **경로 퍼지 수리** (`safeExisting`) | 깨진 파일명·확장자(`.md`→`.py`)를 같은 폴더의 유일 근접 실존 파일로 자동 복구. | 🟡 | `66c9d1c` |
| **editfix** (old_string 관용) | 공백만 다른 old_string을 실제 텍스트에 재매칭해 성공시킴. 실패 시 진짜 파일 내용을 반환. | ✅ | 기존 |
| **서브에이전트 라우팅** (라이브락 가드) | 파일 산출 작업을 읽기전용 explore에 잘못 위임해 무한 루프에 빠지던 것 차단 — code로 유도 + 라이브락 시 중단·반환. | ✅ | `207c534` |
| **language-tutor 하네스** (스킬 규칙) | 날조 방지 게이트(답 없이 평가 금지), 대화형/생성 위임 분리, 출력 언어 규칙(모국어/목표어). | ✅ | harness |
| **/스킬명 발동 · 라우팅** | 슬래시로 하네스 스킬 직접 발동, 도메인 요청을 chat이 아닌 agent로, 코드작업을 critic으로. | ✅ | 여러 커밋 |

## 검증된 레버 · 폐기된 아이디어

> “측정으로 증명 못 하면 기본값으로 넣지 않는다.” 폐기 목록을 남기는 것도 결과다 — 재발명을 막는다.

**검증된 레버**
- **critic 루프 컨텍스트 격리** — 여러 과제에서 일관되게 pass rate 상승. 유일하게 반복 검증된 하네스 레버.
- **AST 편집(재현 회피)** — ①(편집매칭 실패) 제거를 측정으로 확인 (0 vs 2–3).
- **모델 업그레이드 Q8** — ②(재현 스파이럴)를 실질 제거. 진단의 결정적 확증.

**측정 후 폐기**
- **repfix — ② 텍스트 넛지** — ON 1/5 vs OFF 0/5, 노이즈 범위. 말로는 ②를 못 잡음 → 커밋 안 함.
- **auto-gate · escalate-on-spiral** — 이전 측정에서 가치 0 또는 손상된 워크스페이스 상속.
- **2-tier 클라우드 리뷰어** — 정밀 게이트가 있으면 추가 가치 0 → 비활성.

## 인프라에서 배운 것

측정 도중 llama-server(빌드 b9144 + Gemma SWA)가 반복 크래시했다 — 모델이 아니라 서버 버그. 원인과 해결을 분리해 확정했다.

| 증상 | 원인 | 해결 플래그 |
|---|---|---|
| 코어덤프 (`load_tgt`) | prompt-cache 체크포인트 복원 버그 | `--cache-ram 0` |
| 서브에이전트/동시실행 시 재크래시 | 다중 슬롯 KV 캐시(SWA) 버그 | `--parallel 1` |

안정 실행: `-c 32768 --jinja --cache-ram 0 --parallel 1`. 이 조건에서 파이프라인이 크래시 없이 완주.
