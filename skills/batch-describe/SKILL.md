# Batch Describe — 목록 일괄 설명(자기 지식 기반)

리스트(패키지·라이브러리·용어 등)를 읽어 각 항목에 짧은 설명을 붙여 새 파일로 저장하는 작업의 **정해진 절차**다. 이 유형은 "어떻게 설명을 구할까"를 고민하다 삽질하기 쉬우므로, **결정하지 말고 아래 절차를 그대로 따른다.**

## 핵심 원칙 (반드시)

1. **전수 조회하지 마라(삽질 방지).** 500개를 하나씩 `pip show`/PyPI API/웹검색으로 훑지 마라 — 로컬엔 설치도 안 돼 있고, 느린 per-item 조회로 헤매다 무너진다. **대부분의 항목은 네가 이미 안다 → 아는 것은 네 지식으로 즉시 쓴다.**
2. **한 접근에 커밋.** 방법을 재검토하지 마라. 스크립트를 v2/v3로 다시 쓰지 마라. 아래 절차 하나로 끝낸다.
3. **모르는 '소수'만 조회로 보강(계층형 폴백).** 확신이 안 서는 항목만 골라서, 문서 MCP(예: context7)가 연결돼 있으면 그것으로 설명을 가져온다. 그마저 없거나 실패하면 `이름: (설명 불명)`으로 남긴다. ⚠️ 아는 항목까지 조회하지 마라 — 그게 삽질이다.

## 절차

1. **읽기**: 목록 파일을 `read_file`로 읽는다.
2. **정규화**: 각 줄에서 항목 이름만 추출한다(버전·주석·부가정보 제거). 예: `requests   2.31.0` → `requests`, `numpy==1.26` → `numpy`.
3. **설명 작성(자기 지식)**: 각 항목에 대해 한 줄 설명을 직접 쓴다. 형식은 `이름: 한 줄 설명`.
   - 설명은 "무엇을 하는 것인지"를 1문장으로. 버전·설치법은 넣지 않는다.
   - 예: `Flask: 파이썬 경량 WSGI 웹 프레임워크`, `numpy: 수치 계산용 다차원 배열 라이브러리`.
   - 확신이 안 서는 항목은 지어내지 말고 잠시 비워둔다(다음 단계에서 보강).
4. **모르는 항목만 MCP로 보강(선택)**: 3에서 비운 항목이 있고 문서 MCP가 연결돼 있으면, **그 소수만** 조회한다.
   - context7 예: `mcp__context7__resolve-library-id`(이름→라이브러리 ID) → `mcp__context7__get-library-docs`로 개요를 받아 한 줄로 요약.
   - ⚠️ 아는 항목까지 조회하지 말 것. 조회 대상이 많아지면(수십 개↑) 삽질 신호 — 멈추고 아는 만큼만 채운 뒤 나머지는 `(설명 불명)`.
   - 조회해도 안 나오면 `(설명 불명)`.
5. **나눠서 쓰기 (절대 규칙)**: **한 번의 write_file에 전체를 넣지 마라.** 출력 토큰 상한에 걸려 도구 인자(JSON)가 잘리면 서버가 500(파싱 실패)을 뱉고 작업이 통째로 실패한다.
   - **한 번에 최대 50개** 항목만 쓴다. 그 이상이면 무조건 나눈다.
   - **첫 청크만 write_file**(새 파일 생성). ⚠️ write_file은 덮어쓰기라 두 번째부터 write_file을 쓰면 앞 내용이 날아간다.
   - **이어붙이기는 edit_file로**: `old_string`=현재 파일의 **마지막 줄**(고유), `new_string`=그 마지막 줄 + 개행 + 다음 50개. 이렇게 파일 끝에 계속 덧붙인다.
   - ⚠️ old_string은 짧고 정확하게(마지막 줄 하나만). 길게 잡으면 오타 하나로 매칭이 실패한다. 매칭 실패가 2번 연속되면 read_file로 실제 마지막 줄을 다시 확인하라.
   - 항목이 200개면 4번(write 1 + edit 3)에 나눠 쓰는 게 정상이다. "한 방에 끝내려는" 충동을 버려라 — 그게 이 작업의 대표 실패 원인이다.
6. **품질 검증**: 다 쓴 뒤 `run_command`로 결과를 샘플 확인한다(예: `head -n 20 <출력파일>`, `wc -l`).
   - `(설명 불명)`이나 빈 설명이 **대부분**이면 실패다 — 성공으로 선언하지 말고 3번을 다시 채운다.
   - 항목 수와 출력 줄 수가 대략 맞는지 확인한다.
7. **완료 보고**: 만든 파일 경로, 항목 수, 채운 설명 수(+MCP로 보강한 수)를 한 줄로 요약한다.

## 안티패턴 (하지 마라)

- ❌ `pip show`/API로 설명을 가져오려다 "No description available"만 잔뜩 쓰기
- ❌ 아는 항목까지 전부 MCP/API로 조회(조회는 '정말 모르는 소수'에만 — 전수 조회는 삽질)
- ❌ 같은 결정("조회냐 지식이냐")을 여러 번 재논의
- ❌ 한 번의 write_file로 수백 줄을 통째로 생성(깨짐)
- ❌ 검증 없이 "파일 생성됨 = 완료"로 선언
