# Game Design

좋은 게임 설계는 "재밌는 아이디어"가 아니라 **구현자가 바로 만들 수 있는 명세**다. 모호하면 구현이 흔들린다.

## 1단계: 핵심 게임플레이 루프 정의
플레이어가 반복하는 한 문장: **행동 → 피드백 → 보상/위협**.
예) "착지대를 점프로 건너며(행동) 점수가 오르고(보상) 떨어지면 죽는다(위협)."
이 루프가 재미의 80%다. 먼저 이것만 못 박는다.

## 2단계: MVP로 스코프 통제
- **작고 완성 가능한 게임 > 야심차고 깨진 게임.** 핵심 루프 1개만 먼저 명세한다.
- 추가 기능(레벨·아이템·보스)은 "확장" 섹션으로 분리한다.

## 3단계: 구체적 수치로
추상어 금지. 구현·검증 가능하게 숫자로:
- 이동 속도, 점프 높이, 적 스폰 주기, 점수 규칙, 맵/캔버스 크기, 난이도 곡선.

## 4단계: 승/패 조건 명시
도달 가능한 **승리 조건**과 **패배 조건**을 분명히. (플레이테스트가 이걸 검증한다.)

## 출력 형식 (`_workspace/{phase}_game_design.md`)
```markdown
## 컨셉 (1~2줄)
## 핵심 루프
## 조작 (키/마우스)
## 엔티티·상태 (플레이어/적/아이템 + 속성)
## 승리 조건 / 패배 조건
## 화면 구성 (title/playing/gameover)
## MVP 범위
## 확장 아이디어 (선택)
```

## 원칙 (왜)
- **스코프 통제**: 로컬 모델은 큰 게임에서 길을 잃는다. 작은 완성이 큰 미완성보다 가치 있다.
- **수치화**: 숫자가 있어야 구현자가 만들고 플레이테스터가 검증한다.
- **승패 명시**: 도달 불가한 승리 조건은 게임을 망가뜨린다. 처음부터 못 박는다.
