# Git Push 워크플로우 스킬

변경을 **피처 브랜치 → 커밋 → 사전 게이트 → 푸시** 순서로 안전하게 원격에 반영한다.
푸시는 자동 게이트(비밀 스캔·강제 금지·기본 브랜치 금지·검증)를 통과할 때만 실행된다 —
게이트가 막으면 반환된 이유를 읽고 고친 뒤 다시 시도하라. 게이트를 우회하려 하지 마라.

## 절차

1. **상태 확인**: `git status`, `git diff`로 무엇이 바뀌었는지 파악한다.
2. **피처 브랜치**: 지금 브랜치가 `main`/`master`면 **직접 푸시할 수 없다**.
   `git switch -c feature/<짧은-설명>`으로 피처 브랜치를 만든다.
   (이미 피처 브랜치면 그대로 진행)
3. **비밀 점검(직접)**: 커밋 전에 `git diff`에 API 키·토큰·비밀번호가 없는지 스스로 확인한다.
   있으면 `.env`로 빼고 코드에서 제거한다. (게이트가 또 잡지만, 먼저 막는 게 옳다)
4. **커밋**: `git add -A` 후 Conventional Commits 형식으로 커밋한다.
   - `feat:` 새 기능 / `fix:` 버그 / `docs:` / `refactor:` / `test:` / `chore:`
   - 제목 50자 이내·명령형. 필요하면 본문에 '왜'를 한두 줄.
5. **푸시**: `git push -u origin <브랜치>`로 민다.
   - 이때 사전 게이트가 자동 실행된다:
     - 🚫 비밀 감지 → 그 값을 제거하고 다시.
     - 🚫 검증 실패(테스트 빨강) → 코드를 고쳐 초록으로 만든 뒤 다시.
     - 🚫 강제 푸시 → 하지 마라. 정말 필요하면 `--force-with-lease`.
   - 통과하면 확인 없이 자동으로 푸시된다.
6. **보고**: 푸시한 브랜치와 커밋 메시지, (있으면) 원격 URL을 사용자에게 알린다.

## 금지

- `git push --force`/`-f`로 히스토리 덮어쓰기 (게이트가 차단함)
- `main`/`master`에 직접 푸시 (게이트가 차단함)
- 비밀값을 코드/커밋에 넣기
- 게이트 실패를 무시하고 반복 재시도만 하기 — **원인을 고쳐라**
